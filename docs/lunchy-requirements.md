# Lunchy 要件定義

## Context

チームのランチ店選びにおいて、メンバーの好みや店の情報が共有されていないために意思決定が停滞し、結果としてトップダウンの指揮系統に頼らざるを得なくなる問題がある。これは上司・部下双方にとって負担となる。Lunchyはこの問題を解消し、民主的かつスムーズにランチの店が決まるWebサービスを目指す。

## Decisions

### D1: サービスの目的

- **選択肢**: 個人向け店探し / チーム向け意思決定支援
- **決定**: チームのランチ店選びにおける「情報不足」と「トップダウン疲れ」を解消するサービス
- **理由**: チームでの店決めが進まず、トップダウンになりがちな課題を解決する
- **トレードオフ**: 個人利用のユースケースは対象外

### D2: ターゲットユーザー

- **選択肢**: 少人数 / 中規模 / 大規模チーム
- **決定**: 3〜10人のオフィス勤務チーム、ランチに特化
- **理由**: まずは最も一般的なユースケースに絞る
- **トレードオフ**: リモート勤務、飲み会等は対象外

### D3: サービス形態

- **選択肢**: ネイティブアプリ / Webサービス / LINEボット等
- **決定**: Webサービス。共有URLから全員が参加。アカウント登録不要
- **理由**: インストール不要で参加障壁が最も低い
- **トレードオフ**: プッシュ通知等のネイティブ機能は使えない

### D4: コアユーザーフロー

- **選択肢**: 好み入力のみ / 候補提案+投票 / 自由提案
- **決定**: 「好み・気分を選択式で入力 → AIがGoogle Maps連携で候補を3件提示 → 多数決で決定」
- **理由**: シンプルで全員が平等に参加できるフロー
- **トレードオフ**: メンバーによる候補追加は不可

### D5: 好み入力の設計

- **選択肢**: 自由テキスト / 選択式 / 併用
- **決定**: 選択式のみ。ステップ形式で以下の順に回答:
  1. アレルギー（選択）
  2. 肉 / 魚 / それ以外
  3. お腹空いている度（10〜0、0はもう食べたくない）
  4. 食べたい場所（スキップ可）
  5. 予算
- **理由**: 迷わない、タップだけで完了、入力品質が安定する
- **トレードオフ**: 曖昧な気分（「さっぱりしたもの」等）は表現できない。毎回ゼロから入力

### D6: 入力の締め切り

- **選択肢**: 時間制限 / 全員入力で自動進行 / 幹事が手動
- **決定**: 幹事が手動で「締め切り」ボタンを押す
- **理由**: 全員入力待ちだと一人が忘れると進まない。時間制限は柔軟性がない
- **トレードオフ**: 幹事が締め切り忘れるリスクがある

### D7: 候補提示と投票方式

- **選択肢**: 3件 / 5件、決選投票あり/なし
- **決定**: AIが3件の候補を提示。1回目で過半数なければ上位2件で決選投票
- **理由**: 3択はちょうど良い選択肢数。決選投票で確実に過半数で決まる
- **トレードオフ**: 候補が少ないため、全員が満足する店が入らない可能性

### D8: 候補の表示情報

- **選択肢**: 最小限（店名のみ）/ 標準 / 詳細（レビュー含む）
- **決定**: 店名・ジャンル・徒歩時間・評価（星）・価格帯・写真1枚
- **理由**: 比較に必要十分な情報量。多すぎると選択疲れ
- **トレードオフ**: 詳細なレビューや口コミは表示しない

### D9: 好みの衝突時の方針

- **選択肢**: 全員満足を目指す / 多数派優先 / AI判断
- **決定**: 多数派の好みを最優先。少数派向けの候補は出さない
- **理由**: 明確なルールで納得感がある
- **トレードオフ**: 少数派の好みは反映されない

### D10: 投票の匿名性

- **選択肢**: 記名 / 無記名
- **決定**: 無記名。票数のみ表示
- **理由**: トップダウン防止のコンセプトと合致。上司への忖度を排除
- **トレードオフ**: 誰が何を選んだか追跡できない

### D11: 同数時の決定方法

- **選択肢**: ランダム / 幹事判断 / 再投票
- **決定**: ランダムで決定
- **理由**: 再投票は面倒、幹事判断はトップダウンに逆行する
- **トレードオフ**: 運任せになる

### D12: 投票結果の表示タイミング

- **選択肢**: リアルタイム / 一斉公開
- **決定**: 全員投票後に一斉公開
- **理由**: リアルタイム表示は同調圧力を生む
- **トレードオフ**: 投票中は盛り上がりに欠ける

### D13: 幹事の役割

- **選択肢**: 管理者権限あり / 最小限の権限
- **決定**: 幹事の権限はセッション作成と入力締め切りのみ。好み入力・投票には一メンバーとして参加。幹事ローテーション管理は不要
- **理由**: 「場を作る人」であって「決める人」ではない
- **トレードオフ**: 問題が起きた時に幹事が介入できない

### D14: オフィス位置の設定

- **選択肢**: 幹事入力 / GPS / 事前登録
- **決定**: セッション作成時に幹事が住所・場所を入力
- **理由**: GPSはプライバシー懸念、事前登録はアカウント機能が必要
- **トレードオフ**: 毎回入力が必要

### D15: 検索範囲

- **選択肢**: 固定 / 可変
- **決定**: 徒歩10分以内で固定
- **理由**: 設定項目を減らしてシンプルに
- **トレードオフ**: 立地によっては候補が少なくなる/多くなる

### D16: 営業時間チェック

- **選択肢**: 自動チェック / チェックなし
- **決定**: 自動チェック。ランチタイムに営業中の店のみ候補に出す
- **理由**: 閉まっている店が候補に出ると体験が悪い
- **トレードオフ**: Google Mapsの営業時間データが不正確な場合がある

### D17: セッションのライフサイクル

- **選択肢**: 使い捨て / 永続 / 履歴あり
- **決定**: 完全使い捨て。過去の履歴機能はMVPでは実装しない。同じ店が出ることは許容
- **理由**: MVPとしてシンプルさを優先
- **トレードオフ**: 連日同じ店が候補に出る可能性がある

### D18: URL共有方法

- **選択肢**: アプリ内共有機能 / コピーボタンのみ
- **決定**: URLコピーボタンのみ用意。共有手段はユーザーに任せる
- **理由**: Slack/Teams/LINE等、チームごとに使うツールが異なる
- **トレードオフ**: 共有のひと手間がかかる

### D19: 技術スタック

- **選択肢**: Next.js / Vue+別バックエンド / シンプルHTML+JS
- **決定**: Next.js（フロント+API Routes）、Vercel（デプロイ）、Vercel KV（セッションデータ一時保存）。モノリス構成
- **理由**: 小規模サービスに最適。フロント・バックエンド一体化で開発効率が高い。Vercel無料枠でMVPに十分
- **トレードオフ**: Vercelへのロックイン

### D20: AI/地図連携

- **選択肢**: Gemini単体 / Google Maps API単体 / 組み合わせ
- **決定**: Gemini API（Grounding with Google Maps）を使用
- **理由**: API呼び出し1回でAI推論とGoogle Mapsデータ取得を同時に実現
- **トレードオフ**: 新しい機能のため、仕様変更リスクがある

### D21: 対応デバイス

- **選択肢**: スマホ優先 / PC優先 / 両方均等
- **決定**: PC・スマホ両方均等に対応するレスポンシブデザイン
- **理由**: オフィスではPC、移動中はスマホと両方使われる
- **トレードオフ**: 両方のUIを最適化する開発コスト

### D22: 認証

- **選択肢**: アカウント登録 / URL認証 / 認証なし
- **決定**: 認証不要。URLが認証を兼ね、参加時に表示名のみ入力
- **理由**: 参加障壁を最小限にする
- **トレードオフ**: URLが漏れると第三者が参加できる

### D23: 同時利用上限

- **選択肢**: 制限あり / 制限なし
- **決定**: MVPでは考慮しない
- **理由**: まずは動くものを優先
- **トレードオフ**: 大量アクセス時に不安定になる可能性

### D24: 多言語対応

- **選択肢**: 日本語のみ / 多言語
- **決定**: 日本語のみ
- **理由**: ターゲットユーザーが日本のオフィスワーカー
- **トレードオフ**: 海外チームは利用不可

### D25: 投票進捗の表示

- **選択肢**: 進捗表示あり / なし
- **決定**: 幹事の画面に「N/M人投票済み」を表示（好み入力・投票フェーズ両方）。誰が投票したかは非表示（無記名維持）
- **理由**: 幹事が残り人数を把握して締め切り判断できる
- **トレードオフ**: 幹事だけが見える情報格差が生まれる

### D26: Google Maps連携方式

- **選択肢**: Gemini Grounding with Google Maps / Places API + Gemini 2段構成
- **決定**: Gemini API の Grounding with Google Maps を採用
- **理由**: API呼び出し1回でシンプル。MVPとして最速
- **トレードオフ**: 細かいフィルタリング（徒歩10分以内等）の精度はGeminiの解釈に依存
