# UIデザイン詳細

## Context

Lunchy の全10画面のUIデザインを Figma で制作するにあたり、カラーパレット・タイポグラフィ・レイアウト・コンポーネント・インタラクション・エッジケースを事前に決定する。実装前にデザインを固めることで手戻りを減らす。

## Decisions

### D1: トーン&マナー

- **選択肢**: カジュアル・ポップ / クリーン・モダン / フードフォーカス
- **決定**: クリーン・モダンベース、暖色アクセントで「冷たすぎない」バランス
- **理由**: 幅広い年齢層・オフィスの雰囲気に馴染む。業務ツール感と楽しさの両立
- **トレードオフ**: ポップさは控えめになる

### D2: カラーパレット

- **選択肢**: オレンジ系 / コーラル系 / アンバー・マスタード系
- **決定**: オレンジ系アクセント
  - Primary: `#FF6B35`
  - Background: `#FAFAFA`
  - Surface: `#FFFFFF`
  - Text: `#1A1A1A`
  - Success: `#10B981`
  - Error: `#EF4444`
  - Info: `#3B82F6`
  - Gray: `#6B7280` / `#D1D5DB` / `#F3F4F6`
- **理由**: 食欲を刺激する色。CTAボタンとしての視認性が高い
- **トレードオフ**: エラー色（赤）との近さに注意

### D3: タイポグラフィ

- **選択肢**: Inter + Noto Sans JP / Plus Jakarta Sans + Noto Sans JP / システムフォント
- **決定**: システムフォントのみ（`-apple-system, BlinkMacSystemFont, "Segoe UI", ...`）
  - 見出し: 24px / 20px / 16px（bold）
  - 本文: 15px（regular）
  - サブテキスト: 13px（gray）
  - ボタン: 16px（semibold）
- **理由**: 読み込み速度優先。MVPの実用ツールにWebフォントは不要
- **トレードオフ**: デバイス間で見た目が若干異なる

### D4: ボタン・コンポーネントスタイル

- **選択肢**: 角丸大きめ(12px) / 角丸控えめ(8px) / 完全ラウンド
- **決定**: 角丸控えめ（radius: 8px）で全コンポーネント統一
  - Primary: `#FF6B35` 背景、白文字、高さ48px
  - Secondary: 白背景、`#FF6B35` ボーダー+文字
  - Disabled: `#D1D5DB` 背景、`#9CA3AF` 文字
- **理由**: クリーン・モダンのトーンに合致。入力欄・カード・ボタンに統一適用可能
- **トレードオフ**: 特になし

### D5: レイアウト基本構造

- **選択肢**: センターカード型 / PC時サイドバー付き / フルワイド+コンテンツ制約
- **決定**: センターカード型（max-width: 448px、中央配置）
  - コンテナ: `max-w-md mx-auto`
  - パディング: スマホ `px-4`、PC `px-0`
  - 背景: `#F3F4F6`、カード: `#FFFFFF`
- **理由**: 1画面1アクションのシンプルなフロー。実質スマホファーストの1種類で済む
- **トレードオフ**: PCの広い画面を活かせない

### D6: トップページ構成

- **選択肢**: ミニマル / 軽い説明付き / ヒーロー画像付き
- **決定**: 軽い説明付き（ロゴ + キャッチコピー + 3ステップ図解 + CTAボタン）
- **理由**: 初めて使う幹事に「何が起きるか」を伝えて安心感を与える
- **トレードオフ**: ミニマルよりは情報量が多い

### D7: ロゴとキャッチコピー

- **選択肢**: テキストのみ / 絵文字+テキスト / カスタムアイコン+テキスト
- **決定**: 🍽️ 絵文字+テキスト「Lunchy」、キャッチコピー「チームのランチ、みんなで決めよう」
- **理由**: コストゼロで食事サービスであることが伝わる。コピーは民主的なコンセプトを表現
- **トレードオフ**: ブランド感は弱い（MVP後に改善可能）

### D8: 3ステップ図解

- **選択肢**: ユーザー行動ベース / フロー全体ベース
- **決定**: フロー全体ベース
  1. 🔗 URLを共有 —「リンクを送るだけで参加できる」
  2. 📝 好みを入力 —「選ぶだけの簡単回答」
  3. 🎉 お店が決まる —「AI提案 × 投票で民主的に決定」
- **理由**: トップページを見る幹事にとって「メンバーをどう巻き込むか」が最初の関心事
- **トレードオフ**: 特になし

### D9: セッション作成フォーム

- **選択肢**: フリーテキスト / Google Places Autocomplete / フリーテキスト+補足
- **決定**: フリーテキスト +「駅名やビル名でもOK」の補足テキスト
  - 構成: ページタイトル → 表示名入力 → オフィスの場所入力 → 作成ボタン
- **理由**: Gemini Grounding が場所名を柔軟に解釈する前提。Places APIはMVPには過剰
- **トレードオフ**: 入力の曖昧さに依存

### D10: URL共有画面

- **選択肢**: URLコピーのみ / 案内メッセージ付き / ダッシュボードへの導線付き
- **決定**: URL + ダッシュボードへの導線
  - 構成: ✅アイコン +「セッションを作成しました！」→ URL+コピーボタン →「このURLをチームに共有してください」→「ダッシュボードへ」ボタン
- **理由**: コピー後の「次に何をすればいい？」を明確にする
- **トレードオフ**: 特になし

### D11: メンバー参加画面

- **選択肢**: 名前入力のみ / セッション情報付き / 参加済みメンバー表示
- **決定**: 名前入力 + セッション情報（幹事名）
  - 構成: 🍽️ ロゴ →「田中さんのランチセッション」→ 表示名入力 →「参加する」ボタン
- **理由**: 幹事名が見えることで「何に参加するか」が分かり安心
- **トレードオフ**: 特になし

### D12: 好み入力 Step 1 — アレルギー選択

- **選択肢**: 主要7品目 / 7品目+追加 / シンプル4択
- **決定**: トグルチップで特定原材料7品目（卵 / 乳 / 小麦 / えび / かに / そば / 落花生）＋「その他」任意入力欄。未選択かつ未入力＝なし
- **理由**: 日本で最も一般的に記載されている7品目。トグルチップならタップだけで入力負荷が低い。任意入力で漏れもカバー
- **トレードオフ**: 7品目以外のアレルギーは自由入力に依存

### D13: 好み入力 Step 2 — 気分選択

- **選択肢**: 3択 / 4択 / 3択+なんでも
- **決定**: 4択、大きめカード型ボタン（単一選択）
  - 🥩 肉 / 🐟 魚 / 🥗 野菜・ヘルシー / 🍽️ なんでも
- **理由**: 「それ以外」より「野菜・ヘルシー」が意図明確。「なんでも」でこだわりなしの逃げ道
- **トレードオフ**: 複数選択不可（「肉も魚も」は選べない）

### D14: 好み入力 Step 3 — 空腹度

- **選択肢**: スライダー(0-10) / 5段階ボタン / 3段階ボタン
- **決定**: 5段階、大きめカード型ボタン（単一選択）
  - ペコペコ / けっこう空いてる / 普通 / 軽めがいい / ほぼ食べたくない
- **理由**: 言葉で直感的に選べる。3段階より細やかに表現できる
- **トレードオフ**: 5択は若干多い

### D15: 好み入力 Step 4 — 食べたい場所

- **選択肢**: 場所タイプ3択 / 雰囲気ベース4択 / ミックス5択
- **決定**: 3択、大きめカード型ボタン + スキップリンク
  - 🍽️ お店で食べる / 🥡 テイクアウト / 🙆 どちらでも
- **理由**: Google Mapsで「テイクアウト可」は明確なフィルタになる。雰囲気はデータから判別困難
- **トレードオフ**: 雰囲気の好みは反映されない

### D16: 好み入力 Step 5 — 予算

- **選択肢**: 3段階 / 4段階 / 3段階+こだわりなし
- **決定**: 3段階 + こだわりなし、大きめカード型ボタン
  - 💰 〜1,000円（節約モード）
  - 💰💰 1,000〜1,500円（いつも通り）
  - 💰💰💰 1,500円〜（ちょっと贅沢）
  - 🙆 こだわりなし
- **理由**: 1,000円がランチの心理的な区切り。こだわりなしで予算を気にしない人にも対応
- **トレードオフ**: 粗い粒度

### D17: 好み入力 — 進捗インジケーター

- **選択肢**: ステップドット / テキスト / プログレスバー
- **決定**: プログレスバー（オレンジ、画面上部）＋ 戻るボタン（左上「←」）
- **理由**: 5ステップは多いので、視覚的な進捗表示が安心感を与える。戻るボタンで押し間違いに対応
- **トレードオフ**: 特になし

### D18: 入力完了・待機画面

- **選択肢**: シンプル完了 / 完了+人数表示 / 完了のみ（進捗なし）
- **決定**: シンプル完了メッセージ（✅ +「回答完了！幹事が締め切るまでお待ちください」）、静的表示
- **理由**: メンバー側に人数表示は圧力になる。長時間滞在は想定しないため演出は不要
- **トレードオフ**: 待機中は何も情報がない

### D19: 幹事の好み入力タイミング

- **選択肢**: ダッシュボードから誘導 / ダッシュボード内埋め込み / 自動で先に入力
- **決定**: セッション作成 → URL共有 → 好み入力 → ダッシュボードの順で自動遷移
- **理由**: 入力し忘れを防止。ダッシュボード到達時点で幹事は回答済み
- **トレードオフ**: ダッシュボードに辿り着くまでのステップが増える

### D20: 締め切りの確認UI

- **選択肢**: 即実行 / 確認ダイアログ / 条件付き
- **決定**: 条件付き（全員回答済み→即実行、未回答者あり→確認ダイアログ）。締め切り後は「お店を探しています...」ローディング → 自動で投票フェーズへ
- **理由**: 全員回答済みなら確認は無駄。未回答時の誤操作は戻せないので確認が必要
- **トレードオフ**: 条件分岐の実装コスト

### D21: 幹事の投票フェーズ

- **選択肢**: ダッシュボードに進捗のみ / 投票画面と統合 / 投票画面に遷移+進捗バナー
- **決定**: 幹事も通常の投票画面に遷移し、上部に「N/M人投票済み」バナーを追加表示
- **理由**: 幹事も一メンバーとして同じ投票画面を使う。実装がシンプルで体験が統一
- **トレードオフ**: 幹事ダッシュボードと投票画面が分離する

### D22: 投票画面のレイアウト

- **選択肢**: 縦並びカード / 横スワイプ / 縦並びコンパクト
- **決定**: 縦並び大カード（写真大きめ）＋ カードタップで選択 → 下部「投票する」ボタンで確定
- **理由**: 写真を大きく見せて直感的な判断を促す。全件見比べてから確定できる
- **トレードオフ**: スクロールが必要

### D23: 決選投票画面

- **選択肢**: 投票画面と同じ / VS横並び / 縦並び+VS演出
- **決定**: 縦並び2枚カード間に「VS」演出。投票UIは通常投票と同じ（タップ選択→投票ボタン）
- **理由**: 2択の対決感を出しつつ、スマホ375pxでも見やすい
- **トレードオフ**: 特になし

### D24: 結果発表の演出

- **選択肢**: 即表示 / カウントダウン / 投票内訳→結果の順
- **決定**: 即表示
- **理由**: 各自のスマホで非同期に見る状況ではカウントダウンは空回りする。速さがUX
- **トレードオフ**: 盛り上がり演出はない

### D25: 結果画面の勝者カード

- **選択肢**: 投票カードと同じ / 追加情報付き / Google Maps埋め込み
- **決定**: 投票カード + 追加情報（住所・電話番号）。カードタップでGoogle Mapsの店舗ページに遷移
- **理由**: 住所・電話番号で「予約しよう」「場所を確認しよう」に対応。Google Mapsリンクで詳細に飛べる
- **トレードオフ**: カードの情報量が増える

### D26: 投票内訳の表示

- **選択肢**: 水平バーグラフ / 数字のみ / 円グラフ / 表示しない
- **決定**: 表示しない
- **理由**: ユーザー判断。結果画面は勝者カード + Google Mapsリンクに集中
- **トレードオフ**: 「民主的に決めた」実感が薄れる可能性

### D27: エラー状態の表示

- **選択肢**: インラインメッセージ / エラー専用画面 / 使い分け
- **決定**: 使い分け
  - 入力バリデーション（名前空欄等）: インライン赤テキスト
  - 致命的エラー（セッション不在・期限切れ）: 専用画面（⚠️ + メッセージ +「トップページへ戻る」）
- **理由**: エラーの種類に応じた自然な表示
- **トレードオフ**: 2パターンの実装が必要

### D28: ローディング表示

- **選択肢**: スピナー / スケルトンスクリーン / スピナー+テキスト
- **決定**: スピナー + 状況テキスト
  - API取得時:「読み込み中...」
  - AI候補生成時:「お店を探しています...」
- **理由**: テキストで何を待っているか分かり安心
- **トレードオフ**: 特になし

### D29: 画面遷移トランジション

- **選択肢**: なし / フェードイン / スライド等使い分け
- **決定**: トランジションなし。即切り替え
- **理由**: 実装コストゼロ。動作の軽快さがUX貢献。MVPに不要
- **トレードオフ**: 画面遷移が素っ気ない
