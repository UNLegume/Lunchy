# 開発ルール

## Context

Lunchy プロジェクトの開発において、コーディング規約、開発プロセス、Claude Code への指示、品質基準を統一するためのルールを策定する。このドキュメントの決定事項は CLAUDE.md に反映され、Claude Code の振る舞いを制御する。

## Decisions

### D1: TypeScript の厳密さ

- **選択肢**: strict: true / ゆるめ
- **決定**: `strict: true`
- **理由**: 型安全を最大化し、MVPでもバグ防止の恩恵を得る
- **トレードオフ**: 型定義の手間が増える

### D2: コンポーネントの書き方

- **選択肢**: 関数コンポーネント + hooks のみ / 制約なし
- **決定**: 関数コンポーネント + hooks のみ。クラスコンポーネント禁止
- **理由**: Next.js App Router との相性が良い
- **トレードオフ**: なし（現代のReact開発では標準）

### D3: CSS の書き方

- **選択肢**: Tailwind のみ / Tailwind + CSS Modules / Tailwind + CSS-in-JS
- **決定**: Tailwind CSS のユーティリティクラスのみ
- **理由**: シンプルに統一してCSS設計の議論を排除
- **トレードオフ**: 複雑なスタイルの表現が冗長になる場合がある

### D4: ルーティング方式

- **選択肢**: App Router / Pages Router
- **決定**: App Router（`app/` ディレクトリ）
- **理由**: Next.js の最新標準。Server Components が使える
- **トレードオフ**: Pages Router に比べて学習コストがやや高い

### D5: コンポーネント配置ルール

- **選択肢**: 画面固有は app/ + 共通は components/ / すべて components/ / Atomic Design
- **決定**: 画面固有のコンポーネントは `app/` 配下に同居、共通コンポーネントは `components/` に分離
- **理由**: MVPの規模でAtomic Designは過剰
- **トレードオフ**: 規模が大きくなると整理が必要になる

### D6: 命名規則

- **選択肢**: 役割ごとに慣例 / すべて kebab-case
- **決定**: コンポーネントは PascalCase（`SessionCard.tsx`）、ユーティリティは camelCase（`formatDate.ts`）、ディレクトリは kebab-case（`session-create/`）
- **理由**: 最も一般的な Next.js の規約に従う
- **トレードオフ**: なし

### D7: ブランチ戦略

- **選択肢**: GitHub Flow / Git Flow / トランクベース
- **決定**: GitHub Flow（`main` + feature ブランチのみ）
- **理由**: MVP開発にはシンプルで十分。個人〜少人数開発に最適
- **トレードオフ**: リリースブランチがないため複雑なリリース管理は不可

### D8: ブランチ命名規則

- **選択肢**: feature/タスク番号-説明 / タスク番号/説明 / 自由
- **決定**: `feature/タスク番号-短い説明`（例: `feature/03-session-create`）
- **理由**: タスク番号と紐づくので追跡しやすい
- **トレードオフ**: なし

### D9: コミットメッセージ形式

- **選択肢**: Conventional Commits / 自由 / タスク番号プレフィックス
- **決定**: Conventional Commits（`feat:`, `fix:`, `chore:` 等）
- **理由**: 将来 CHANGELOG 自動生成等にも使える
- **トレードオフ**: コミット時のひと手間

### D10: PRルール

- **選択肢**: PR + セルフマージ / PR + レビュー必須 / 直接push
- **決定**: feature ブランチから main へ PR → セルフマージOK
- **理由**: 一人開発の場合、レビュー必須は形骸化する
- **トレードオフ**: レビューがないためバグが混入しやすい

### D11: 作業委託方針

- **選択肢**: グローバルルール踏襲 / 直接実装OK
- **決定**: グローバルルール踏襲。Opus 指揮、Sonnet 実装、Explore 調査
- **理由**: 一貫性を保ち、メインコンテキストをクリーンに維持
- **トレードオフ**: サブエージェント起動のオーバーヘッド

### D12: タスクドキュメント参照

- **選択肢**: 必ず参照 / 必要に応じて
- **決定**: 実装開始前に必ず `docs/task/XX-*.md` を読んでから着手する
- **理由**: 要件と実装の乖離を防ぐ
- **トレードオフ**: 毎回の読み込みコスト

### D13: 要件定義との整合性チェック

- **選択肢**: 必ず確認 / チェックしない
- **決定**: 画面実装時に `docs/lunchy-requirements.md` の該当 Decision を確認してから実装する
- **理由**: D番号を意識して実装することで要件との乖離を防ぐ
- **トレードオフ**: 確認の手間

### D14: テスト方針

- **選択肢**: API必須+UI任意 / すべて必須 / テスト不要
- **決定**: API Routes は必ずテストを書く。UIコンポーネントのテストは任意
- **理由**: MVPの速度とバックエンドの信頼性のバランス
- **トレードオフ**: UIのリグレッションは手動確認に頼る

### D15: Linter / Formatter

- **選択肢**: pre-commit hook / CI / 手動
- **決定**: pre-commit hook で OxLint + Prettier（高速チェック）。push 時に ESLint + Prettier を自動実行（厳密チェック）
- **理由**: ローカルでは速度重視、push時に厳密チェックで品質担保
- **トレードオフ**: ツール設定の初期コスト

### D16: `any` 型の使用

- **選択肢**: 完全禁止 / やむを得ない場合許容 / 自由
- **決定**: 完全禁止。外部APIレスポンス等でやむを得ない場合は AskUserQuestion でユーザーに確認してから使用
- **理由**: 型安全を最大限に保つ
- **トレードオフ**: 外部APIの型定義に手間がかかる

### D17: console の使用

- **選択肢**: 完全禁止 / error のみ許容 / 自由
- **決定**: `console.log` / `console.warn` / `console.error` すべて本番コードでは禁止（ESLint `no-console` を error）
- **理由**: ログはログ基盤で管理すべき（MVP後に検討）
- **トレードオフ**: デバッグ時に不便

### D18: 未使用の import / 変数

- **選択肢**: error / warning
- **決定**: ESLint で error にする
- **理由**: 不要なコードは即削除
- **トレードオフ**: なし

### D19: 環境変数管理

- **選択肢**: .env.local + .env.example / Vercel設定のみ
- **決定**: `.env.local` で管理。`.env.example` にキー名のみ記載してコミット
- **理由**: 新しい開発者がすぐに必要な環境変数を把握できる
- **トレードオフ**: なし

### D20: 環境変数バリデーション

- **選択肢**: 起動時チェック / 都度チェック / チェックなし
- **決定**: アプリ起動時に必須環境変数の存在チェック。なければエラーで起動失敗
- **理由**: 設定漏れを早期発見
- **トレードオフ**: なし

### D21: APIエラーレスポンス形式

- **選択肢**: 統一フォーマット / Next.js デフォルト
- **決定**: 統一フォーマット `{ error: { code: string, message: string } }` + 適切な HTTP ステータス
- **理由**: フロント側でのエラー処理が楽になる
- **トレードオフ**: エラーレスポンスの実装コスト

### D22: ユーザー向けエラー表示

- **選択肢**: やさしいメッセージ / 詳細表示
- **決定**: ユーザーにはやさしいメッセージ、詳細はサーバーログのみ
- **理由**: 本番でスタックトレースを見せない
- **トレードオフ**: ユーザーが問題を自己解決しづらい

### D23: ログ基盤

- **選択肢**: カスタムロガー / 外部サービス / なし
- **決定**: MVP ではログ基盤は作らない。Vercel のデフォルトログで対応
- **理由**: MVP の速度を優先
- **トレードオフ**: 本番障害時のデバッグが困難

### D24: セッションURL生成

- **選択肢**: UUID v4 / カスタムトークン
- **決定**: UUID v4 で生成
- **理由**: 十分にランダムで推測不可能
- **トレードオフ**: URLが長い

### D25: レートリミット

- **選択肢**: 実装する / しない
- **決定**: MVP では実装しない
- **理由**: Vercel 側にある程度の保護がある
- **トレードオフ**: 悪意あるリクエストに弱い

### D26: 入力値サニタイズ

- **選択肢**: サーバー側サニタイズ / React のエスケープに任せる
- **決定**: サーバー側でサニタイズ（XSS防止）。React のエスケープとの二重防御
- **理由**: `dangerouslySetInnerHTML` 等で穴が開く可能性を防ぐ
- **トレードオフ**: サニタイズ処理の実装コスト

### D27: スコープ厳守

- **選択肢**: 厳密に守る / 柔軟に対応
- **決定**: 要件定義にない機能追加、タスク外の作業、ユーザー確認なしのパッケージ追加は禁止
- **理由**: スコープの肥大化を防ぐ
- **トレードオフ**: 柔軟性が下がる

### D28: ドキュメントとの乖離時の対応

- **選択肢**: ユーザーに確認 / ドキュメント更新 / 実装側で判断
- **決定**: 実装を止めて AskUserQuestion でユーザーに確認する
- **理由**: 勝手に解釈して進めない
- **トレードオフ**: 確認待ちで作業が止まる

### D29: リファクタリング

- **選択肢**: 指示された場合のみ / 自主的にOK
- **決定**: タスクとして明示的に指示された場合のみ行う
- **理由**: 「ついでにリファクタ」はスコープクリープの原因
- **トレードオフ**: 技術的負債が蓄積する可能性

### D30: タスク完了条件

- **選択肢**: チェックリストベース / コードが動くだけ
- **決定**: コードが動く + テスト通過（APIの場合）+ タスクドキュメントのチェックリストが全て完了
- **理由**: チェックリストベースで管理し、完了基準を明確に
- **トレードオフ**: 完了までの手間が増える

### D31: タスク完了時のドキュメント更新

- **選択肢**: ステータス更新する / しない
- **決定**: `docs/task/XX-*.md` の status を `completed` に更新し、完了日を記録
- **理由**: 進捗が一目でわかる
- **トレードオフ**: 更新の手間

### D32: 実装中の課題・疑問の記録

- **選択肢**: ドキュメントに集約 / TODO コメント / 両方
- **決定**: 該当ドキュメントの Implementation Q&A に記録 + AskUserQuestion でユーザーに確認。TODO コメントは使わない
- **理由**: ドキュメントに集約して散逸を防ぐ
- **トレードオフ**: ドキュメント更新の手間

### D33: 実装サイクル

- **選択肢**: 自由 / 固定サイクル
- **決定**: 必ず「探索 → 計画 → 実装 → 検証」のループを回す。探索（コードベース調査・ドキュメント確認）、計画（実装方針をユーザーに提示・承認）、実装（計画に基づくコード実装）、検証（テスト・動作確認・ドキュメント整合性チェック）。各ステップをスキップしない
- **理由**: 実装の品質を担保する基本サイクル
- **トレードオフ**: 実装速度が落ちる

### D34: 最低限の実装

- **選択肢**: 最低限のみ / 便利機能も追加OK
- **決定**: 最低限の機能のみ実装。タスクドキュメントと要件定義に記載された範囲を超えない。「あると便利」な機能、将来のための拡張、過剰なエラーハンドリング、不要な抽象化は一切行わない
- **理由**: MVP の速度を最大化し、スコープクリープを防ぐ
- **トレードオフ**: 後から機能追加が必要になる
