# テスト方針

## バックエンド（API Routes）: TDD 必須

テスト駆動開発で進める。以下のサイクルを厳守する。

```
Red   → テストを先に書く（失敗することを確認）
Green → テストが通る最小限のコードを書く
Refactor → コードを整理する（テストが通り続けることを確認）
```

### TDD の進め方

1. タスクドキュメントの仕様からテストケースを洗い出す
2. テストを1件書く → 実行して **Red**（失敗）を確認
3. そのテストだけを通す最小限の実装を書く → **Green**
4. 必要に応じてリファクタリング → テストが通ることを確認
5. 次のテストケースへ戻る（2へ）

### テスト対象

- 全 API Routes のリクエスト/レスポンス
- バリデーション（不正入力の拒否）
- ビジネスロジック（多数決集計、過半数判定、ランダム決定等）
- エラーレスポンスの形式（`{ error: { code, message } }`）
- KV ヘルパー関数

## フロントエンド: コンポーネント駆動 + Playwright ビジュアルチェック

### 実装の進め方

コンポーネント単位でボトムアップに構築する。

```
1. 共通コンポーネントを単体で作成（ボタン、カード、入力フィールド等）
2. コンポーネントを組み合わせてセクション・ブロックを構成
3. セクションを組み合わせてページを構成
```

### Playwright ビジュアルチェック（必須）

ページに影響する変更を行った場合、必ず Playwright MCP でスクリーンショットを撮り、表示崩れがないか確認する。

```
コード変更 → Playwright でページを開く → スクリーンショット撮影
  → 崩れなし → 次へ進む
  → 崩れあり → 修正 → 再度スクリーンショット → 崩れが解消するまでループ
```

#### チェック対象

- スマホ（375px）とPC（1280px）の両方で確認
- 対象となる「ページに影響する変更」:
  - コンポーネントの新規追加・修正
  - レイアウト・スタイルの変更
  - データ表示内容の変更
  - レスポンシブ対応の調整

#### 崩れの判断基準

- 要素の重なり・はみ出し
- 意図しない余白・隙間
- テキストの切れ・折り返し異常
- ボタン・入力欄のサイズ崩れ
- Figma デザインとの明らかな乖離
